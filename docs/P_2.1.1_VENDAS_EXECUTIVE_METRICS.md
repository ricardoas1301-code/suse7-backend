### CARD 3 — Vendas em R$

| Regra | Definição oficial |
|-------|-------------------|
| Nome técnico | `gross_sales_brl` |
| Definição | Soma do valor **bruto** das linhas válidas no período |
| Coluna base | `sales_order_items.gross_amount` (enriquecido ML quando existir snapshot) |
| Formato API | `string` decimal `"6309.33"` |
| Cálculo | **Backend Decimal.js** — proibido frontend |

**Incluir (vendas válidas):** pagas, concluídas, entregues, em andamento financeiramente válidas, “A combinar” se pedido ML válido.

**Excluir:** canceladas, devolvidas integralmente, pagamento recusado, pedidos teste/inválidos.

**Gap atual:** summary soma **todas** as linhas sem regra de status.

---

### CARD 4 — Quantidade de Vendas

| Regra | Definição oficial |
|-------|-------------------|
| Nome técnico | `orders_count` |
| Definição | Contagem de **pedidos** (`sales_order_id` DISTINCT) válidos no período |
| Unidades (extra) | `items_quantity_sold` = Σ `quantity` das linhas válidas |

**Gap atual:** `orders_count` existe no summary, mas sem filtro de status/período.

---

### CARD 5 — Lucro R$

| Regra | Definição oficial |
|-------|-------------------|
| Nome técnico | `contribution_profit_brl` |
| Definição | Margem de contribuição agregada (lucro bruto operacional) |
| Fórmula (por linha, depois Σ) | `net_received − custo_produto − impostos_internos − operação − embalagem − outros_custos_persistidos` |
| Margem | `contribution_margin_percent` = `contribution_profit_brl / gross_sales_brl × 100` (Decimal, 2 casas) |
| Tooltip UI | *"Lucro bruto estimado pela margem de contribuição: valor recebido menos custo do produto, impostos internos, operação, embalagem e outros custos cadastrados."* |

**Regras:** nunca calcular no frontend; reutilizar `computeSaleDetailRealResult` (ou materializar por linha no sync).

**Gap atual:** summary **não** expõe lucro; lista usa fórmula simplificada.

---

### CARD 6 — Conversão

| Regra | Definição oficial |
|-------|-------------------|
| Nome técnico | `sales_conversion_rate_percent` |
| Fórmula | `orders_count / visits_count × 100` |
| `visits_count` | **Não existe** no backend hoje |
| Comportamento até existir tráfego | `sales_conversion_rate_percent: null`, `conversion_data_status: "unavailable"`, UI `"—"` / *Sem dados* |

**Estados `conversion_data_status`:**

| Valor | Condição |
|-------|----------|
| `available` | `visits_count > 0` e pedidos válidos |
| `partial` | visitas parciais (ex.: só alguns anúncios) — definir na integração ML traffic |
| `unavailable` | sem fonte de visitas |

## 9. Riscos

| Risco | Impacto | Mitigação P_2.1.2 |
|-------|---------|-------------------|
| Summary usa `Number` / float | Drift centavos em totais | RPC + Decimal.js |
| Lucro lista ≠ lucro Raio-X | KPI inconsistente | Agregar com pipeline do detail |
| Sem filtro de status | KPI inflado com canceladas | Tabela de status válidos ML + filtro SQL |
| `get_catalog_rankings` fora do repo | Ranking produto vazio em prod | Migration versionada + testes |
| `listing_sales_metrics` all-time | Rankings ignoram período | Agregação ad hoc ou MV por período |
| Visitas inexistentes | Card Conversão sempre vazio | Integração ML metrics / fase posterior |
| `filter` UI-only | Chips enganam o seller | Implementar no BE ou desabilitar até P_2.1.2 |
| Performance (scan grande) | Timeout Vercel | RPC paginada, índices `(user_id, sale_date)`, cap + `truncated_scan` |

## 13. P_2.1.2 — implementação backend

**Endpoint:** `GET /api/sales/executive-summary`  
**Handler:** `src/handlers/sales/executiveSummary.js`  
**Agregação:** `src/domain/sales/buildSaleExecutiveSummary.js`

**Query params suportados:** `marketplace`, `marketplace_account_id`, `seller_company_id`, `start_date`, `end_date`, `period_preset` (`all`, `today`, `7d`, `30d`, `month`, `custom`), `filter`, `q`, `ranking_limit` (max 10).

**Lucro agregado:** `computeSaleDetailRealResult` + Decimal.js por linha; imposto via `resolveSaleInternalTaxProfile` (cache por empresa/conta).

**Conversão:** `conversion_data_status: "unavailable"` até existir `visits_count`.

**Limite:** `S7_EXECUTIVE_SUMMARY_MAX_ORDERS` (default 5000) → `truncated_scan` + warning em `data_quality`.
